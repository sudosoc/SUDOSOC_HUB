Basic Scanning
nmap 192.168.1.1Basic scan of a single host (top 1000 ports)
nmap 192.168.1.0/24Scan entire subnet
nmap -iL targets.txtScan targets from a file
nmap 192.168.1.1-254Scan range of IPs
nmap -p 22,80,443 hostScan specific ports
nmap -p- hostScan all 65535 ports
nmap -p 1-1000 hostScan port range 1-1000
nmap --top-ports 100 hostScan top 100 most common ports
Scan Techniques
nmap -sS hostSYN scan (stealth, default with root) — most common
nmap -sT hostTCP connect scan (no root required)
nmap -sU hostUDP scan (slow but important)
nmap -sA hostACK scan — firewall rule mapping
nmap -sN hostNULL scan — no flags set
nmap -sF hostFIN scan
nmap -sX hostXmas scan — FIN, PSH, URG flags
nmap -sW hostWindow scan
nmap -sO hostIP protocol scan
nmap -sn 192.168.1.0/24Ping scan / host discovery only (no port scan)
nmap -Pn hostSkip ping, treat all hosts as up

Timing & Performance
nmap -T0 hostParanoid — very slow, minimal IDS detection
nmap -T1 hostSneaky — slow scan
nmap -T2 hostPolite — reduces bandwidth
nmap -T3 hostNormal (default)
nmap -T4 hostAggressive — faster, requires reliable network
nmap -T5 hostInsane — fastest, likely to miss results or be detected
nmap --min-rate 1000 hostSend at least 1000 packets per second
nmap --max-retries 1 hostOnly retry once (faster but less accurate)

DNS Filters

dns.qry.type == 1A record queries

tcp.flags.syn==1 && tcp.flags.ack==0Port scan detection (SYN without ACK)
icmp.type == 8 && icmp.code == 0ICMP echo requests (ping sweep)

frame.len > 1500Jumbo frames or oversized packets
!(ip.src == 192.168.0.0/16)Traffic not from RFC1918 — external traffic
tcp.dstport in {4444 8080 9090 1337}Common backdoor/C2 ports

IR Phases (NIST SP 800-61)
1. PreparationEstablish IR team, policies, tools, and communication plans before an incident
2. Detection & AnalysisIdentify indicators of compromise, determine scope and severity
3. ContainmentShort-term (isolate systems), Long-term (patch/segment)
4. EradicationRemove malware, close entry points, reset credentials
5. RecoveryRestore systems, monitor closely, verify normal operation
6. Lessons LearnedDocument timeline, update playbooks, improve detection
Initial Triage (First 15 Minutes)
Identify alert sourceSIEM alert? EDR? User report? Email? Network alert?
Determine severityCritical / High / Medium / Low based on asset criticality + IOC confidence
Preserve evidenceTake memory dump before rebooting or isolating. Memory is volatile!
Document everythingTimestamp every action. Use UTC. Who, what, when, how.
Notify stakeholdersEscalate per runbook — manager, legal, CISO if critical
Check false positiveVerify IOC against threat intel before taking action

Containment Decision Matrix
Isolate from networkIf active exfiltration or C2 communication detected
Block IP/domain at firewallKnown C2 IPs/domains from threat intel
Disable user accountIf compromised credentials confirmed
Reset passwordsAll accounts on compromised system + shared credentials
Revoke API keys/tokensIf secrets were exposed in logs or processes
Snapshot VM before wipingPreserve forensic evidence for legal/insurance
Do NOT reboot immediatelyVolatile evidence (RAM, processes, network connections) will be lost
Alert Triage Process
1. ValidateIs this a true positive or false positive? Check context, asset info, user behavior baseline
2. EnrichAdd threat intel context: VirusTotal, Shodan, AbuseIPDB, internal CMDB
3. ScopeIs this isolated or widespread? Check lateral movement, other affected systems
4. ClassifyAttack type: phishing / malware / brute force / insider / DDoS / exfiltration
5. RespondFollow runbook for alert type. Document all actions with timestamps
6. CloseDocument resolution, update ticket, tune detection if FP

T1566 — PhishingWatch for: unusual sender, suspicious attachments, links to uncommon domains
T1059 — Command InterpreterPowerShell, cmd, bash with obfuscated commands or unusual parent processes
T1078 — Valid AccountsLegitimate creds used maliciously: off-hours login, new geo-location, bulk access
T1055 — Process InjectionProcesses injecting into lsass, explorer, svchost
T1110 — Brute ForceMultiple 4625 events, high frequency auth failures from same source
T1071 — C2 via WebBeaconing pattern: regular intervals, unusual User-Agent, encrypted POST to rare domain
T1048 — ExfiltrationLarge DNS TXT responses, unusual data out via HTTP/FTP/cloud storage
T1547 — Boot PersistenceNew registry Run keys, scheduled tasks, services added
T1003 — Credential DumpLSASS access, SAM dump, Mimikatz artifacts
T1021 — Lateral MovementSMB/WMI/RDP connections between internal hosts, PsExec, WMIExec

IOC Types & Confidence
File Hash (SHA256)High confidence — unique to a file. Don't rely on MD5 alone (collision risk)
IP AddressMedium — IPs rotate. Check ASN, hosting provider, geolocation context
DomainMedium — DGA domains are random. Check WHOIS, cert, registration date
URLMedium-High — more specific than domain. Check full path
Email AddressLow-Medium — easily spoofed. Check SPF/DKIM/DMARC headers
User AgentLow — trivial to change. Useful for correlation only
Mutex / Registry KeyHigh — often hardcoded in malware families
Network SignatureHigh — protocol anomalies, beacon timing, Yara network rules
